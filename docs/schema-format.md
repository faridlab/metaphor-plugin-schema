# Schema Format Reference

This document covers all schema file formats, types, attributes, and syntax supported by the Metaphor schema system.

## File Types

| Extension | Purpose | Format |
|-----------|---------|--------|
| `*.model.yaml` | Entity definitions, shared types, enums | YAML (recommended) |
| `*.hook.yaml` | Entity lifecycle behaviors, state machines, rules, triggers | YAML |
| `*.workflow.yaml` | Multi-step business processes (Saga pattern) | YAML |
| `*.model.schema` | Entity definitions (legacy) | Custom DSL |
| `*.workflow.schema` | Workflow definitions (legacy) | Custom DSL |

Schema files are discovered automatically in the module's `schema/` directory:

```
libs/modules/{module}/schema/
  ├── user.model.yaml
  ├── user.hook.yaml
  ├── order.model.yaml
  ├── order.hook.yaml
  ├── order_processing.workflow.yaml
  └── index.model.yaml        # Module-level configuration
```

## YAML Model Format

The `*.model.yaml` file is the primary schema definition format. It supports the following top-level sections:

```yaml
# Data definitions
models:           # Model/table definitions
enums:            # Enumeration types
types:            # Custom reusable type definitions
generators:       # Per-file generator configuration

# DDD extensions (see ddd-guide.md)
entities:         # Enhanced models with behavior
value_objects:    # Domain-specific composite/wrapper types
domain_services:  # Services with dependencies
event_sourced:    # Event sourcing configurations
authorization:    # RBAC/ABAC permissions and roles
usecases:         # Application layer operations
events:           # Domain event definitions
projections:      # CQRS read models
services:         # Application services
handlers:         # Event handlers
subscribes_to:    # Cross-module event subscriptions
integration:      # ACL adapter definitions
presentation:     # HTTP/gRPC config
dtos:             # Request/response definitions
versioning:       # API versioning config
traits:           # Repository trait definitions
```

### Models

Define data models that map to database tables and Rust structs:

```yaml
models:
  - name: Order
    description: Customer order with line items and lifecycle status
    collection: orders          # Database table name
    soft_delete: true           # Enable soft-delete (adds deleted_at field)
    fields:
      id:
        type: uuid
        primary_key: true
      customer_id:
        type: uuid
        required: true
      email:
        type: string
        required: true
        unique: true
      status:
        type: OrderStatus       # Reference to an enum
        required: true
      total:
        type: decimal
      metadata:
        type: json
      created_at:
        type: timestamp
        auto: true              # Auto-generated by database
      updated_at:
        type: timestamp
        auto: true
    relations:
      customer:
        type: Customer
        attributes: ["@one", "@foreign_key(customer_id)", "@on_delete(restrict)"]
      items:
        type: OrderItem[]
        attributes: ["@one_to_many", "@on_delete(cascade)"]
    indexes:
      - fields: [customer_id]
      - fields: [status, created_at]
      - fields: [email]
        unique: true
```

> **Strict parsing.** Model entries reject unknown top-level keys.
> Misplacing per-model generator gating (`disabled: [...]` written at
> the model root instead of under `config.generators.disabled`) now
> fails parse with `unknown field 'disabled', expected one of …`
> instead of being silently ignored. See
> [Generators Configuration](#generators-configuration) for the correct
> shape.

### Field Properties

| Property | Type | Description |
|----------|------|-------------|
| `type` | string | Field type (primitive, enum reference, or custom type) |
| `required` | bool | Field cannot be null (default: `false`) |
| `primary_key` | bool | Mark as primary key (default: `false`) |
| `unique` | bool | Add unique constraint (default: `false`) |
| `auto` | bool | Auto-generated value (default: `false`) |
| `default` | any | Default value for the field |

### Enums

Define enumeration types:

```yaml
enums:
  - name: OrderStatus
    variants:
      - Pending
      - Confirmed
      - Shipped
      - Delivered
      - Cancelled

  - name: UserRole
    values: [admin, editor, viewer]    # Alternative short syntax
```

Enum names are automatically converted to **PascalCase** in generated Rust code:

```yaml
# Schema
enums:
  - name: relationship_type
    values: [lead, customer, partner]

# Generated Rust
# pub enum RelationshipType { Lead, Customer, Partner }
```

### Shared Types

Define reusable type compositions:

```yaml
types:
  - name: AuditFields
    fields:
      created_at:
        type: timestamp
        auto: true
      updated_at:
        type: timestamp
        auto: true
      created_by:
        type: uuid
      updated_by:
        type: uuid
```

### Generators Configuration

Control which generators run at the file level:

```yaml
generators:
  enabled: [rust, sql, repository]     # Whitelist mode
  disabled: [cqrs, projection]         # Blacklist mode
  cqrs: true                           # Explicit opt-in for a specific target
```

**Resolution rules:**

- `enabled` and `disabled` are mutually exclusive — pick one mode per file.
- Per-target booleans (`cqrs: true`, `event-store: false`) override the
  `enabled` / `disabled` lists, so you can use a list as the baseline
  and toggle individual targets on top.
- Target names match the canonical names from the per-platform docs
  (e.g. [generate-rust.md](generate-rust.md) target tables). Aliases
  (`gql` for `graphql`, `migration` for `sql`, etc.) are accepted.

**Cross-target side effects:**

Some Rust targets emit wiring that depends on whether sibling targets
were generated. The `module` target (which writes `lib.rs`) tracks the
effective enabled set and **drops** the router-mount, gRPC service
registration, and `MergedObject!` entries when `grpc` or `graphql` is
disabled — so the module still compiles when only a subset of
transports is in use. The full list of cross-target dependencies is in
[generate-rust.md → module](generate-rust.md#module----module-wiring-librs--modrs).

**Per-model overrides:**

Individual model entries accept the same `enabled` / `disabled` lists,
either as a direct `generators:` field or wrapped under `config:` to
mirror the file-level shape:

```yaml
models:
  audit_log:
    generators:
      disabled: [cqrs, graphql]   # skip CQRS + GraphQL for this one model
    fields: { ... }

  read_only_view:
    config:
      generators:
        enabled: [rust, sql]      # only emit the data layer for this entity
    fields: { ... }
```

When both forms appear on the same entry, the direct `generators:` field
wins. Models that opt out of a target are dropped wholesale from that
target's output — no file is written and no entry appears in the
generated parent `mod.rs`.

See the per-platform docs for the full target catalog:

| Platform | Target list |
|----------|-------------|
| Rust server | [generate-rust.md § Generation Targets](generate-rust.md#generation-targets) |
| Kotlin mobile | [generate-kotlin.md § Targets](generate-kotlin.md) |
| Webapp (TS+React) | [generate-webapp.md § Targets](generate-webapp.md) |

## Primitive Types

The schema system supports 30 primitive types with automatic mapping to Rust, SQL, Protocol Buffer, and Kotlin types:

| Type | Aliases | Rust | PostgreSQL | Proto |
|------|---------|------|-----------|-------|
| `string` | `str` | `String` | `TEXT` | `string` |
| `int` | `integer` | `i32` | `INTEGER` | `int32` |
| `int32` | `i32` | `i32` | `INTEGER` | `int32` |
| `int64` | `i64`, `long` | `i64` | `BIGINT` | `int64` |
| `float` | `f64` | `f64` | `DOUBLE PRECISION` | `double` |
| `float32` | `f32` | `f32` | `REAL` | `float` |
| `float64` | - | `f64` | `DOUBLE PRECISION` | `double` |
| `bool` | `boolean` | `bool` | `BOOLEAN` | `bool` |
| `bytes` | - | `Vec<u8>` | `BYTEA` | `bytes` |

## Field Attributes

Used in the legacy DSL format and in relation `attributes` arrays:

| Attribute | Description | Example |
|-----------|-------------|---------|
| `@id` | Primary key | `id uuid @id` |
| `@unique` | Unique constraint | `email string @unique` |
| `@required` | Not nullable | `name string @required` |
| `@default(value)` | Default value | `@default(now)`, `@default(uuid)`, `@default(active)` |
| `@min(n)` | Minimum length/value | `@min(3)` |
| `@max(n)` | Maximum length/value | `@max(255)` |
| `@pattern(regex)` | Regex validation pattern | `@pattern([a-z0-9-]+)` |
| `@foreign_key(ref)` | Foreign key reference | `@foreign_key(User.id)` |
| `@soft_delete` | Enable soft-delete on model. On the Kotlin side, when paired with an `@audit_metadata` field, the entity gains a derived `val isDeleted: Boolean get() = metadata["deleted_at"] != null` helper. If the schema also declares an explicit `is_deleted` column, the helper is suppressed (the column wins) so there is exactly one canonical `isDeleted` property. | - |
| `@hashed` | Field is stored hashed (e.g., passwords) | `password string @hashed` |
| `@audit_metadata` | JSONB audit metadata field. On the Kotlin side this is emitted as the `Metadata` typealias (`Map<String, JsonElement?>`), not raw `JsonElement` — see [generate-kotlin.md](./generate-kotlin.md). | - |
| `@omit_if_none` | Omit this nullable field from the serialized response when its value is `None` (emits `#[serde(skip_serializing_if = "Option::is_none")]`). By default nullable fields serialize as explicit `null` for a stable response shape; use this attribute to opt out per field. | `deleted_at datetime? @omit_if_none` |
| `@private` | Field-level security: the field is pruned from serialized responses for non-owner/non-root callers. Collected into the generated `EntityRepoMeta::private_fields()` override (as camelCase response keys); backbone-core's `apply_field_security` strips these keys at the handler. | `ssn string @private` |
| `@owner` | Marks the tenant/ownership column used by field-level security to decide who counts as the owner. Emitted as the `EntityRepoMeta::owner_field()` override (camelCase response key). At most one per model. | `user_id uuid @owner` |

### Foreign Key Actions

| Attribute | Values | Description |
|-----------|--------|-------------|
| `@on_delete(action)` | `cascade`, `restrict`, `set_null`, `set_default` | Action when parent is deleted |
| `@on_update(action)` | `cascade`, `restrict`, `set_null`, `set_default` | Action when parent is updated |

## Relation Attributes

| Attribute | Description |
|-----------|-------------|
| `@one` | One-to-one relationship |
| `@many` | One-to-many relationship |
| `@one_to_many` | One-to-many (explicit) |
| `@many_to_many` | Many-to-many relationship |
| `@belongs_to` | Inverse relationship |
| `@references(column)` | Custom referenced column |
| `@join_table(name)` | Many-to-many join table name |
| `@join_fk(column)` | Many-to-many FK column |
| `@foreign_key(column)` | Explicit FK column name |

### Relation Examples

```yaml
relations:
  # One-to-one
  profile:
    type: Profile
    attributes: ["@one"]

  # One-to-many with cascade delete
  items:
    type: OrderItem[]
    attributes: ["@one_to_many", "@on_delete(cascade)"]

  # Many-to-many with custom join table
  roles:
    type: Role[]
    attributes: ["@many_to_many", "@join_table(user_roles)", "@join_fk(user_id)"]

  # Belongs-to with foreign key
  customer:
    type: Customer
    attributes: ["@belongs_to", "@foreign_key(customer_id)", "@on_delete(restrict)"]
```

## Indexes

In YAML format:

```yaml
indexes:
  - fields: [email]                    # Single-field index
    unique: true
  - fields: [status, created_at]       # Composite index
  - fields: [customer_id]
```

In legacy DSL format:

```
indexes {
    @@index(email)
    @@index(status, created_at)
    @@unique(resource, action)
}
```

### Partial Indexes

Use the `@where` attribute to emit a partial index — the predicate is appended
verbatim as a SQL `WHERE` clause:

```yaml
indexes:
  - fields: [email]
    unique: true
    attributes: ['@where(deleted_at IS NULL)']
```

**Audit-metadata rewriting.** When the model carries an `@audit_metadata`
JSONB field, audit keys (`created_at`, `updated_at`, `deleted_at`,
`created_by`, `updated_by`, `deleted_by`) are stored *inside* that JSONB
column, not as real columns. To keep the predicate readable, bare
identifiers matching those keys are rewritten to their JSONB form
(`(metadata->>'deleted_at')`). String literals and identifiers already
qualified (e.g. `t.deleted_at`) are left alone, and a model that declares
one of those names as a real column shadows the rewrite.

```yaml
# Authored:
attributes: ['@where(deleted_at IS NULL)']

# Emitted (model has @audit_metadata field `metadata`):
WHERE (metadata->>'deleted_at') IS NULL
```

## Soft Delete

Enable soft-delete on a model to automatically generate:
- A `deleted_at` timestamp field
- Trash listing endpoints
- Restore endpoint
- Repository methods with soft-delete filtering

```yaml
models:
  - name: User
    collection: users
    soft_delete: true
    fields:
      # ... deleted_at is added automatically
```

A model is treated as soft-deletable in two ways:

1. **Explicit** — `soft_delete: true`.
2. **By convention (Backbone)** — the model carries an audit `metadata` field or
   a `deleted_at` field. The Backbone backend exposes the trash endpoints for
   exactly these entities, so the generator infers soft-delete automatically and
   emits the trash use cases.

Soft-deletable entities gain the trash surface in the generated webapp runtime:
`listDeleted`, `restore`, `emptyTrash` (`DELETE /empty`), and `permanentDelete`.
A plain `delete` performs the soft delete itself, so there is no separate
`softDelete` operation.

## Hook YAML Format

The `*.hook.yaml` files define entity lifecycle behaviors:

```yaml
hooks:
  OrderHooks:
    entity: Order

    states:
      - name: pending
        initial: true
        on_enter:
          - action: send_notification
            params: { template: "order_created" }
      - name: confirmed
      - name: shipped
      - name: delivered
        final: true
      - name: cancelled
        final: true

    transitions:
      - from: pending
        to: confirmed
        guard: "has_payment"
        roles: [admin, manager]
      - from: confirmed
        to: shipped
        roles: [admin, warehouse]
      - from: shipped
        to: delivered
      - from: [pending, confirmed]
        to: cancelled
        roles: [admin, customer]

    rules:
      minimum_order_amount:
        when: [create, update]            # create | update | delete (optional)
        condition: "order.total >= 10.00"
        message: "Order total must be at least $10.00"
        code: "ORDER_BELOW_MINIMUM"       # optional — warnings often omit it
        severity: error                   # error | warning (optional)

    permissions:
      customer:
        allow:
          - read                          # shorthand: bare action string
          - action: update                # full form: condition + field scope
            if: "order.status == 'pending'"
            only: [items, shipping_address]
        deny:
          - action: update
            except: [total]               # all fields except these
      admin:
        allow:
          - all

    computed:
      - name: total
        expression: "sum(items.price * items.quantity)"
        type: decimal

    triggers:
      - event: on_transition
        from: pending
        to: confirmed
        action: charge_payment
      - event: on_enter
        state: shipped
        action: send_shipping_notification
```

A trigger may also group several actions under a named key with an optional
`if` condition. Each entry in `actions:` is either a bare action string or the
struct form (`action:` / `type:`, optionally with `params:`) — both spellings
are accepted:

```yaml
    triggers:
      notify_warehouse:
        if: "order.requires_shipping"
        actions:
          - send_shipping_notification          # bare string
          - action: reserve_inventory           # struct form
            params: { warehouse: "default" }
          - type: log_event                      # `type` is an alias for `action`
```

Both the rich map-based form shown above and the canonical list/sequence
authoring form are accepted everywhere `*.hook.yaml` is consumed. The webgen
generator tries the map form first and falls back to the canonical flexible
parser, so it accepts exactly the same hook grammar as the backend codegen — an
authored hook parses in every generator regardless of which spelling it uses.

## Workflow YAML Format

The `*.workflow.yaml` files define multi-step business processes following the Saga pattern:

```yaml
workflows:
  OrderProcessing:
    description: "End-to-end order fulfillment"
    steps:
      - name: validate_order
        action: validate
        entity: Order
        on_failure: abort

      - name: reserve_inventory
        action: reserve
        entity: Inventory
        compensation: release_inventory

      - name: charge_payment
        action: charge
        entity: Payment
        compensation: refund_payment

      - name: ship_order
        action: ship
        entity: Order
        compensation: cancel_shipment

    on_complete:
      - send_confirmation_email
    on_failure:
      - notify_support
```

## Legacy DSL Format

The legacy `.model.schema` format uses a custom DSL syntax. It is still supported for backward compatibility but YAML is recommended for new schemas.

### Model Definition

```
model User {
    collection: users

    fields {
        id          uuid        @id @default(uuid)
        email       email       @unique @required
        username    string      @unique @required @min(3) @max(50)
        password    string      @required @min(8)
        status      Status      @default(active)
        created_at  datetime    @default(now)
        updated_at  datetime?
        deleted_at  datetime?
    }

    relations {
        profile     Profile     @one
        roles       Role[]      @many_to_many
    }

    indexes {
        @@index(email)
        @@index(username)
        @@index(status, created_at)
    }
}
```

### Enum Definition

```
enum Status {
    active      @default @label("Active User")
    inactive    @label("Inactive User")
    pending     @label("Pending Verification")
    suspended   @label("Suspended Account")
}
```

### Workflow Definition (Legacy)

```
workflow OrderApproval {
    states {
        pending     @initial
        approved
        rejected    @final
    }

    transitions {
        pending -> approved [role: manager]
        pending -> rejected [role: manager]
    }

    rules {
        rule minimum_amount {
            condition: "order.total >= 100"
            error: "Orders below $100 do not require approval"
        }
    }
}
```

### Key Differences from YAML

| Feature | Legacy DSL | YAML |
|---------|-----------|------|
| Nullable types | `datetime?` suffix | Omit `required: true` |
| Attributes | Inline `@attr` syntax | `attributes` array or field properties |
| Enums | `enum Name { ... }` block | `enums:` list with `variants` |
| Relations | Inline in `relations { }` block | `relations:` mapping |
| DDD extensions | Not supported | Full support (entities, VOs, services, etc.) |
| Shared types | Not supported | `types:` section |
